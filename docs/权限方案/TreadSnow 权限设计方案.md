#### 权限设计

0-4个权限等级，0没有权限，1个人权限，2部门权限 ，3 部门及下级部门权限 ，4 组织权限

1.界面

在角色的权限界面添加一个tab页，原来那个权限改名为菜单权限，新增的tab为数据权限

界面大概是这样

| 表名 | 读      | 写      | 删除    |
| ---- | ------- | ------- | ------- |
| 会员 | 0-4等级 | 0-4等级 | 0-4等级 |
| 宠物 | 0-4等级 | 0-4等级 | 0-4等级 |
| 附件 | 0-4等级 | 0-4等级 | 0-4等级 |

- 表名列表最好支持多语言

- 0-4等级，点一次+1，再点则回到0 ，样式你看能不能做成dynamics crm那种控件

- 匹配完的数据以json的形式保存到ConfigJson中

  

#### 数据方案

1.查询控制：方法一开始就要查询数据查询用户-角色-权限数据，多个角色去重取最大权限，

- 如果当前entity读权限为0，则返回空对象。
- 如果当前entity读权限为1，则返回负责人为当前登录用户数据+负责团队里面有自己的数据
- 如果当前entity读权限为2，则查询本部门的所有用户和团队，查询负责人为本部门的所有用户数据+负责团队join团队表where部门=本部门
- 如果当前entity读权限为3，则查询本部门以及下级部门的所有用户和团队，查询负责人为本部门以及下级部门的所有用户数据+负责团队join团队表where部门=本部门以及下级部门
- 如果当前entity读权限为4，则查询所有数据

以上条件都要带上租户这个条件，因为是多租户系统

2.编辑控制：和查询方法处理一样，只是判断编辑权限，然后把编辑弹出界面的保存禁用/启用

3.删除控制：和查询方法处理一样，只是判断删除权限，然后把列表删除按钮隐藏/显示



二重防呆：

entity保存和编辑方法需要在后台添加一样的权限校验。
