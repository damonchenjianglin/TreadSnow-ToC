# TreadSnow 数据权限系统实现计划

## Context

实现类似 Dynamics CRM 的数据层级权限系统。当前系统只有菜单级别的权限（ABP标准的Permission体系），缺少数据行级别的权限控制。需要新增部门、团队等组织结构，并基于角色配置每个实体表的读/写/删除权限等级（0-4级），在后端查询和操作时自动过滤数据。

## 权限等级说明

| 等级 | 含义 | 数据范围 | 颜色 | 图标说明 |
|------|------|----------|------|----------|
| 0 | 无权限 | 返回空 | 灰色 `#d9d9d9` | 空心圆/禁止图标 |
| 1 | 个人 | 负责人=当前用户 + 负责团队包含当前用户 | 绿色 `#52c41a` | 单人图标 |
| 2 | 部门 | 本部门所有用户和团队的数据 | 蓝色 `#1890ff` | 部门图标 |
| 3 | 部门及下级 | 本部门+下级部门所有用户和团队的数据 | 橙色 `#fa8c16` | 树形图标 |
| 4 | 组织 | 所有数据（租户内） | 红色 `#f5222d` | 地球/组织图标 |

### 阶段四：后端核心 — 数据权限过滤服务

#### 4.1 新建 `DataPermissionService`（应用服务内部工具类）
- **文件**: `src/TreadSnow.Application/DataPermissions/DataPermissionService.cs`
- 注册为Transient依赖注入
- 核心方法:
  - `GetEffectivePermissionAsync(string entityName)` — 返回当前用户对指定实体的合并后最大权限等级(读/写/删)
  - `ApplyReadFilterAsync<T>(IQueryable<T> query, string entityName)` — 根据读权限等级自动过滤查询
  - `CheckWritePermissionAsync(string entityName, Guid? ownerId, Guid? ownerTeamId)` — 校验写权限
  - `CheckDeletePermissionAsync(string entityName, Guid? ownerId, Guid? ownerTeamId)` — 校验删除权限
- 内部逻辑:
  1. 获取当前用户所有角色（AbpUserRoles + TeamRoles中团队包含当前用户的角色）
  2. 获取每个角色的 RoleDataPermission 配置
  3. 对同一实体的多角色配置取最大等级
  4. 根据等级构建过滤条件:
     - 等级0: 返回空
     - 等级1: Where(OwnerId == currentUserId || OwnerTeamId in 用户所在团队)
     - 等级2: Where(OwnerId in 本部门用户 || OwnerTeamId in 本部门团队)
     - 等级3: Where(OwnerId in 本部门及下级用户 || OwnerTeamId in 本部门及下级团队)
     - 等级4: 不过滤

#### 4.2 修改现有 AppService — 集成数据权限

## 缓存方案建议

对于设计文档中提出的Redis缓存问题：

- **后端**: 使用ABP的 `IDistributedCache<T>` 缓存用户合并后的数据权限，缓存键为 `DataPermission:{UserId}`，过期时间5分钟。角色权限变更时主动清除相关用户缓存。
- **前端**: 登录后调用一次 `GetCurrentUserPermissionsAsync` 获取当前用户的合并权限，存储在Angular Service的内存中，用于UI展示控制（按钮显隐、保存禁用等）。
- **不需要浏览器持久化缓存**（localStorage等），内存即可，每次登录/刷新页面时重新获取。Redis访问在有缓存的情况下只在缓存失效时触发，不会频繁。

## 实施顺序

1. **阶段一 + 阶段二**（Domain + EF Core）— 先建好数据结构，生成迁移
2. **阶段三**（Application 层基础CRUD）— 提供API接口
3. **阶段四**（数据权限核心逻辑）— 实现过滤和校验
4. **阶段五**（前端权限配置UI）— 数据权限配置入口
5. **阶段六**（前端部门/团队管理）— 组织架构管理页面

---

## 验证方式

1. 运行 `dotnet build TreadSnow.sln` 确认编译通过
2. 运行 `dotnet ef migrations add` 生成迁移
3. 运行 `dotnet run --project src/TreadSnow.DbMigrator` 应用迁移
4. 运行 `cd angular && npm run start` 启动前端
5. 在角色管理页面验证数据权限配置弹窗（包括等级颜色说明展示）
6. 创建测试数据，验证不同权限等级下的数据过滤效果
